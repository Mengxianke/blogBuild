为何要做异步控制? ​

隐藏的复杂度: Promise为异步任务，是允诺/承诺。至于promise何时解析, 响应时间是未知。 这就面临着两个问题:

  1. 任务与任务之间存在依赖关系, 任务A与任务B之间存在依赖关系。任务B需要等待任务A解析后，在执行任务B。如果两者没有依赖关系，A与B任务可以并发执行。
  2. 时序问题，任务A早于任务B提交，任务A的结果应该早于任务B呈现。
    导航渲染，上需要对接UI，即用户的操作，下需要对接MapKit和对接C++ Native侧。两者都是返回promise。如何正确处理以上两个问题，变得尤为重要。

对象状态控制 ​

导航渲染建模中, 聚合根需要负责对整个对象状态的维护。 保证状态的一致性。数据的完整性。 完整性和一致性需要借助并发控制实现。

任务编排, 串行异步任务队列 ​

为何要实现异步任务队列?
任务体现在模型的行为上，以及在门面类中，封装的方法上。 队列有先进先出的特性，保证任务与任务之间的先后关系。这种任务的依赖关系体现在类与类的约束关系上。
组合是典型的类间约束关系。根据导航中渲染物的UML类图，路线作为聚合根，它与各种渲染物之间都构成组合关系。 这种父子生命周期的约束关系，需要通过异步任务队列进行维护。 借助PLimit, 将Concurrency设置成1, 变成一个串行化的任务队列。 任务队列会实现以下特性:

  1. 将会依次按顺序执行任务，后一个任务将会等待前一个任务执行完毕，获取结果后，才会执行下一个任务。
  2. 超时机制, 防止任务中出现异常，导致Promise无法解析或拒绝。进而产生队列阻塞的现象。

网络请求竞态 ​

为何要对网络请求加异步控制?
对于一些网络请求，我们希望当本次开始请求时，将不再接收到上一次的请求结果。 否则，由于请求的并发性，时序得不到保证，会出现先发出去的请求, 之后才收到结果。 时序的错乱会导致UI与用户的操作不一致的现象。
在规划结果页面, 当用户请求当前的规划路线，需要把上一次的规划路线结果"取消掉"， 在底图上不显示上一次规划路线的结果。
针对导航中的网络请求，借助PCancelable，实现一个只响应最新请求结果的http并发控制。 并发控制将会实现以下特性。

  1. 保证时序的正确性
  2. 响应最新的网络请求结果。 对之前的网络请求，抛出异常，表明上次请求已经被取消。

            })
        });
        this.creatingDeferredPromise = {
            promise: p,
            reoslve: _resolve,
            reject: _reject
        };

        // 省略业务代码
    }

        // 从defer对象中获取到reject方法

        if (reject) {
            reject();

        this.transitToBeforeRemoveState();
    }

        this.transitToBeforeRemoveState();  
        return false;
    }

}
