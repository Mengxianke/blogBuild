Log日志分析与预期结果: ​

规划结果有三条路线 每一条路线针对某一个事件，应该挂载一个监听器。理论上， 应该只有三个监听器。但从Log日志, 发现监听器的数量有6个(是正常数量的2倍)。结束导航/离开规划页面，监听器依旧 生效，发生内存泄漏。

问题排查: ​

监听器的挂载是在类RoadVehicleRouteLine的构造函数中, 需要排查创建具体实现类CarDrivingRouteLine的过程。
当前只有在remove方法中取消监听, 是否CarDrivingRouteLine的实例从内存中回收前，并没有走remove方法??

ts
// CarDrivingRouteLine的创建类
export class CarDrivingRouteLineBuilder implements RouteLineBuilder {
    // 初始化一次, 在堆中开辟空间
    private carDrivingRouteLine: CarDrivingRouteLine = new CarDrivingRouteLine();

    init() {
        // 变量被替换, 引用消失。作为垃圾，等待被回收，但没有调用remove()方法取消监听。
        this.carDrivingRouteLine = new CarDrivingRouteLine();
    }


}

// 创建者主管类
export class RouteLineDirector {

初步原因分析: ​

remove()方法不是类的析构函数, 当实例被替换后，并没有调用remove()方法。事件中心, 对监听器强引用, 阻碍gc过程。导致内存泄漏。

根因分析 ​

代码review:

ts
export class RoadVehicleRouteLine extends RouteLine {
    protected naviLine?: RoadVehicleNaviLine;

    constructor() {
        //...构造函数
    }
     // 设置路线
    public setNaviLine(naviLine: RoadVehicleNaviLine) {
        this.naviLine = naviLine;
        //... 调用监听器
    }
}

真正设置路线是调用routeLine下的setNaviLine方法, 应该在这个地方，挂载事件监听。 监听器挂载时机错误，导致了此次内存泄漏。

思考 ​

监听器挂载时机错误，通过代码走查的方式不易发现。如何才能更好的阻止此类问题发生。或者提供兜底策略?
在初步原因分析中, 事件中心(EventCenter)强引用了监听器，阻碍了监听器的回收gc。能否修改为弱引用?
详细请看内存管理中事件中心的设计。
