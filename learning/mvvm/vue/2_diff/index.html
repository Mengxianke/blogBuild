MVVM的视图更新，是由数据进行驱动，Model的变化反应到视图上的变化。

            counter++;
        }

在观察者设计模式中， 被观察者通过维护观察者列表，以及通过调用notify()方法，直接调用观察者上的方法。达到传递消息的目的。 主动调用notify()的方法是 在Object.defineProperty对 用户的写操作进行拦截的过程中。响应式数据通过通过setter的方法调用其关联的Dep管理类的notify方法。

Dep类中的notify方法，遍历观察者列表，执行观察者watcher的update方法

js
export default class Dep { 

    notify(info?: DebuggerEventExtraInfo) {
        for (let i = 0, l = subs.length; i < l; i++) {
            const sub = subs[i];
            sub.update();
        }
    }
}

run方法中执行了get()函数，通过调用getter函数来获取，响应式变量的当前值。 在上述的vue组件中，counter变量使用在模板中，对应的是render watcher. 在上一个章节中， 我们知道render watcher的构造器的getter函数是updateComponent()函数，

js
updateComponent = () => {

vm._render生成数据变化后的vNode虚拟结点，将vNode传入 vm._update()中。 我们看一下组件的_update()方法。 preVnode不存在做第一次的渲染。当preVnode节点已经存在， 将会调用vm的__patch__方法。函数入参是前后两个vNode结点。该方法对DOM进行打补丁。进而实现DOM树的更新。

Vue.prototype._update = function (vnode: VNode, hydrating?: boolean) {

    const prevVnode = vm._vnode
    // ..... 
    if (!prevVnode) {
      // initial render
      vm.$el = vm.__patch__(vm.$el, vnode, hydrating, false /* removeOnly */)
    } else {
      // updates
      vm.$el = vm.__patch__(prevVnode, vnode)
    }

}

Vnode ​

Vnode是Vue描述DOM节点的对象。Vue为何采用虚拟节点对DOM节点进行描述?

  1. 跨平台性

  2. 性能考量 不采用Vnode节点，势必就要通过直接操作DOM节点，来直接修改DOM节点(例如本文开篇提及到的vue组件)， 如果每一次对DOM进行真实操作，都会让浏览器重新计算本节点与其字节点的布局位置。 会造成大量的重排，造成性能问题。因此Vue希望通过对比新旧VNode节点，寻找可复用的节点，以及做最小程度上的更新。减少对DOM的真实操作，提高性能。

我们在上一章中了解到，Vue在执行_render函数前，会将template模板通过ast抽象语法树生成render渲染函数。 在_render()函数中，会使用render渲染函数以及createElement方法，生成vNode节点。我们需要先了解VNode的数据结构 在VNode数据结构中:

  1. tag: 表示节点的标签名
  2. data: 包含节点的属性、事件、样式信息
  3. children: 一个数组，Arry类型，里边存放该节点下的所有子节点
  4. text: 字符串，表示节点的文本内容。
  5. elm: DOM元素，表示节点对应的真实DOM节点。

在有了对DOM节点的对象描述后，我们要看一下Vue的比对规则是什么。 设计一个算法的过程中，需要考量其时间复杂度。 打"补丁"的这个算法，本质上是希望，通过最少的更新步骤，能够将一颗VNode节点树转化成另一颗节点树。 这与我们熟知的编辑距离的问题非常相近。通过最少的步骤，插入删除替换，来转化到另一个字符串。 对于VNode虚拟节点，也有删除，删除一个节点，插入, 修改三个操作。 树的编辑距离，可以查看论文: A Robust Algorithm for the Tree Edit Distance 2011年，A Robust Algorithm for the Tree Edit Distance，这个算法将树的编辑距离控制在了O(n ^ 3)的复杂度上。

为此，Vue为了降低时间复杂度，采用了同层比较，深度优先的策略:
同层比较的过程中。如果两个节点对比后发现类型发生了变化，直接销毁该节点，将其替换为新节点。否则认为其是一个可以复用的节点。 在销毁的过程中，其旧节点的子节点一并销毁。不会再参与到对比的流程中。判断是否是相同节点的规则:

  1. key值一样
  2. tag一样
  3. 注释节点
  4. 都定义了data
  5. 当标签是input标签，input标签的type类型必须相同。
js
function sameVnode(a, b) {
  return (
    a.key === b.key &&
    a.asyncFactory === b.asyncFactory &&
    ((a.tag === b.tag &&
      a.isComment === b.isComment &&
      isDef(a.data) === isDef(b.data) &&
      sameInputType(a, b)) ||
      (isTrue(a.isAsyncPlaceholder) && isUndef(b.asyncFactory.error)))
  )
}

在两者VNode都是可复用的节点时，通过深度优先遍历，递归的给当前节点打补丁。 这个过程是深度优先遍历的过程。保证子节点通过打补丁的方式，转化成了新的Vnode节点。

patchVNode(
    oldVnode,
    vnode,
    insertedVnodeQueue,
    ownerArray,

    removeOnly?: any) {
        // 旧节点的孩子节点
        const oldCh = oldVnode.children
        // 新节点的孩子节点
        const ch = vnode.children
        // 比较文本
        if (isUndef(vnode.text)) {
            // 如果新旧节点的孩子相同，对孩子进行可复用性操作。将旧节点的孩子变成新节点的孩子
        if (isDef(oldCh) && isDef(ch)) {
            if (oldCh !== ch)
            updateChildren(elm, oldCh, ch, insertedVnodeQueue, removeOnly)
        } else if (isDef(ch)) {
            // 新节点有孩子，旧节点没有，直接添加
            if (__DEV__) {
            checkDuplicateKeys(ch)
            }
            if (isDef(oldVnode.text)) nodeOps.setTextContent(elm, '')
            addVnodes(elm, null, ch, 0, ch.length - 1, insertedVnodeQueue)
        } else if (isDef(oldCh)) {
            // 旧节点有孩子，新节点没有，删除旧节点的孩子
            removeVnodes(oldCh, 0, oldCh.length - 1)
        } else if (isDef(oldVnode.text)) {
            nodeOps.setTextContent(elm, '')
        }
        } else if (oldVnode.text !== vnode.text) {
            // 文本不同，替换问题
            nodeOps.setTextContent(elm, vnode.text)
        }
    }

updateChildren()方法
定义新旧节点的头尾四个指针，通过头尾对比进行查找可复用的节点。找到可复用节点后，通过移动节点进行复用节点。 找到可复用节点后，向左右中心靠拢。这个算法的关键点是，通过四个指针的移动，来保证旧节点移动时候，它在新节点序列中的相对位置正确。
当四个指针指向的节点均无法复用的情况下，维护一张哈希表。key: 为旧的Vnode的节点的key, 对应的同层中所处的index序列为value. 遍历新节点，尝试通过哈希表找到其对应的旧节点的位置，进行复用。如果没有找到可复用节点，需要创建节点。

  1. 初始化四个指针
js
let oldStartIdx = 0 // 老vnode遍历的开始下标
let newStartIdx = 0 // 新vnode遍历的开始下标

let oldEndVnode = oldCh[oldEndIdx] // 老vnode列表最后一个子元素

let newStartVnode = newCh[0] // 新vnode列表第一个子元素

  1. 两两比较，寻找可复用节点。 如下分为四种情况:

    1. 新旧节点的start节点可以复用。给这两个节点打补丁patchVNode, 深度优先遍历，访问新旧节点。递归遍历结束后，指针同时右移动。

      假设当前旧节点的数据.... [e, .... f]..... oldStart指向节点e, oldEnd指向节点f 假设当前新节点的数据....... [q, .... e]..... newStart指向节点q, newEnd指向节点e oldStartVnode与newEndVnode发现可以复用，需要挪动e节点。oldEnd指向的f, 当前未找到可复用节点。f的位置有几种推测:
      1. f节点位置不可能在新序列的q节点前。注意当前newStart的指针位置，因为如果f的位置在q前，说明f节点已经找到可复用的位置，并发生了移动。那么当前 旧节点的oldEnd指针不可能还指向f节点，相互矛盾。
      2. f节点的位置不可能在新序列中e节点的位置后，同(1)论述，如果在节点e后，注意newEnd的指针位置，如果f在e后，说明f节点已经找到可复用的位置， 并发生了移动。那么当前旧节点的oldEnd指针不可能指向f节点，相互矛盾。
      3. f节点只能在新序列的[q, ...., e]中间, 如果f在[q, e]中间，如果挪动e的位置，e节点只能在f节点之后，并且只能是移动到旧序列f节点的下一个位置。 为什么只能是当前旧序列f节点的下一个位置。假设当前旧序列中[e, ..... f, a]。节点f后已经有了一个节点a, 这个节点a已经是复用节点，并且移动到f节点后。其在新序列中的相对位置也是节点a在节点f后，并且是在新序列节点e还要之后。只有这种情况，节点a才能发生复用性移动。
